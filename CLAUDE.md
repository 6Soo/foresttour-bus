# foresttour-bus

숲길따라 감성여행 도구 모음. GitHub Pages로 배포됩니다 (bus.foresttour.kr).
빌드 과정 없는 순수 정적 사이트입니다 (HTML/CSS/JS).

## 구조

| 파일 | 역할 |
|---|---|
| `index.html` + `script.js` + `styles.css` | 버스 좌석표 도구 (기존) |
| `itinerary.html` + `itinerary.js` + `itinerary.css` | 여행 일정 직접 입력·공유 페이지 (토스 스타일 모바일 UI) |
| `import.html` + `import.js` + `import.css` | 이미지(OCR)·텍스트 → 일정 자동 변환 도구 |
| `parse-rules.js` | **두 일정 페이지가 공유**하는 카테고리 정의(`CATS`)와 파싱 규칙(`PARSE_RULES`) |

- 일정 데이터는 URL 해시(`#d=base64`)로 공유되고 localStorage(`foresttour-itinerary-v1`)에 자동 저장됩니다.
- import 페이지의 결과는 `itinerary.html#d=...` 로 넘겨서 열립니다.
- 버스 페이지와 일정 페이지는 디자인 통일성을 유지할 필요가 없습니다.

## ⭐ 일정 이미지 학습 워크플로 (중요)

사용자는 다음카페에 올리는 일정 이미지(캡처)를 수시로 공유합니다.
**새 일정 이미지를 받으면 아래 순서로 "학습"하세요:**

1. 이미지를 읽고 일정 양식을 파악한다 (일차 표기, 숙박 표기, 항목 유형, 날짜 표기).
2. `parse-rules.js`의 규칙으로 파싱되는지 검토한다:
   - `PARSE_RULES.dayHeaders` — 일차 시작 패턴 (예: `첫날..`, `<제1일>`, `DAY 1`, `1일차`)
   - `PARSE_RULES.stayInline/stayAnywhere/stayBrokenOpen/stayKeyword` — 숙박 패턴 (예: `[숙박- ...]`, `[OO온천호텔]`, 여는 대괄호 유실)
   - `PARSE_RULES.categoryKeywords` — 항목 분류 키워드
   - `PARSE_RULES.dateHints` — 제목 속 날짜 추출 (예: `7월4(토)~9/야마가타`)
3. 새 양식이면 패턴/키워드를 **추가**한다 (기존 패턴은 유지 — 옛 이미지도 계속 파싱돼야 함).
   OCR 오인식 변형도 함께 등록한다 (예: `둘쨋날` → OCR이 `둘짤`로 읽음).
   **⚠️ 줄바꿈을 기계적으로 믿지 말 것**: 일정 이미지는 모바일 화면 캡처라 원문 한 줄이
   화면 폭 때문에 여러 줄로 꺾여 있다 (예: `트레킹(1시간/2` + `시간)`).
   줄을 새 항목으로 만들지/앞 줄에 이어붙일지는 **문맥으로 판단**해야 한다 —
   괄호가 안 닫혔는지, 닫는 기호로 시작하는지, 영문 단어가 이어지는지 등
   (`isContinuationLine` 참고). 새 꺾임 유형을 발견하면 여기에 규칙을 추가한다.
   단, 병합 규칙은 반드시 "언제 멈출지"(일차 시작, 숙박 대괄호, 병합 줄 수 제한)를
   함께 정의할 것 — 쓰레기 문자 하나에 문서 전체가 붙는 사고가 실제로 있었다.
4. 새 항목 유형이 필요하면 `CATS`에 카테고리를 추가한다 (이모지/색상 포함).
   `CATS`는 두 페이지 모두에서 자동 반영된다.
5. 필요하면 일정 카드 디자인(itinerary.css)도 함께 개선한다.
6. 테스트: `node`로 파서 단위 테스트 후 Playwright(사전 설치된 `/opt/pw-browsers/chromium`)로 페이지 확인.
   ```bash
   node -e "const {parseScheduleText}=require('./parse-rules.js'); console.log(JSON.stringify(parseScheduleText(\`첫날..테스트\`),null,2))"
   ```

### ⚠️ 사용자 교정은 하드코딩이 아니라 근본 원인 수정으로

사용자가 인식/분류 오류를 교정해주면 (예: "반 시게루의 그 온천호텔"이 "시게루의 온천호텔"로
인식됨), **그 사례만 예외 처리하지 말 것.** 반드시:

1. 왜 잘못됐는지 원인을 추적한다 (어느 규칙/필터가 어떤 논리로 그 결과를 만들었는지).
   예: 위 사례의 원인은 노이즈 필터가 "1~2자 + 의미문자 2개 미만" 토큰을 지우는 규칙이라
   완성형 한글 단어 `반`, `그`, `후`까지 삭제한 것.
2. 같은 **유형**의 오류가 재발하지 않도록 규칙 자체를 고친다
   (예: 완성형 한글 1~2자 토큰은 보존하도록 필터 조건 수정).
3. 고친 규칙이 기존 테스트(3개 양식 + 실제 이미지 E2E)를 깨지 않는지 확인한다.
4. 원인과 수정 내용을 이 문서의 학습 기록에 남긴다.

### ⚠️ 애매한 지명 경계는 웹 검색으로 확인

OCR 결과에서 인접한 두 줄이 별개 항목인지 하나의 항목인지 애매하면 (예: "마루이케" / "16나한상"),
**추측하지 말고 시간이 걸리더라도 웹 검색으로 실제 관계를 확인**한다:

1. 지명을 검색해 실존 여부·위치·상호 관계(같은 코스인지, 인접 명소인지)를 파악한다.
2. 한 항목으로 확인되면 `PARSE_RULES.compoundPlaces`에 근거 주석과 함께 등록한다
   (예: 丸池様와 十六羅漢岩은 유자마치의 ~2km 세트 코스 → `['마루이케', '16나한상']`).
3. 사용자가 작성하는 일정의 표기 습관도 규칙화한다
   (예: '점심'/'저녁' 뒤에 메뉴·상호명을 붙여 씀 → `mealSplit`으로 별도 항목 분리).

### 학습 기록 (사용자 교정 → 규칙 수정 사례)

- "반 시게루의 그 온천호텔" → "시게루의 온천호텔" 인식: 노이즈 필터가 1~2자 토큰을
  일괄 삭제한 것이 원인 → 완성형 한글 1~2자 토큰은 보존하도록 규칙 수정.
- 트레킹/산책 구분 (사용자 요청): "요코테성 호숫길"은 산책이지 본격 트레킹이 아님 →
  `walk`(산책 🚶) 카테고리 신설. `트레킹/등산/하이킹` 단어가 있으면 trek,
  `산책/~길(호숫길·숲길·생태길 등)/습원`은 walk. '야생화길 트레킹'처럼 둘 다 있으면
  trek이 우선 (키워드 순서).
- 항목 끝의 `&` 등 매달린 연결 기호 제거 (사용자 교정: "무라카미성터&").
- 식사 표기 규칙 (사용자 확인): '점심'/'저녁' 뒤에 메뉴·상호명을 붙여 씀 →
  다른 일정과 한 줄에 붙어 있으면 식사부터 별도 항목으로 분리 (`mealSplit`),
  OCR 쓰레기에 붙은 '점심'도 표기는 살림.
- "마루이케 16나한상"은 한 항목 (웹 검색으로 확인: 丸池様·十六羅漢岩은 유자마치 세트 코스)
  → `compoundPlaces` 사전 신설.
- "SASHIMAKI습원" → OCR이 "SASHIMAKIE"로 읽음: 라틴+한글이 한 단어에 붙으면 OCR이
  토큰 전체를 영문으로 판단해 한글부가 붕괴되는 유형. 검색으로 실존 지명(刺巻湿原,
  아키타 센보쿠 미즈바쇼 군락지) 확인 후 `ocrFixes` 교정 사전 신설. 같은 유형 발견 시
  검색으로 확인하고 여기에 추가할 것.

### 사용자 교정 학습 (런타임)

import 미리보기에서 사용자가 분류를 고치면 `localStorage('foresttour-learned-cats-v1')`에
`{kw, cat}` 형태로 저장되어 다음 분석 때 키워드 규칙보다 우선 적용됩니다 (`classifyText` 참고).

### 지금까지 학습한 이미지 양식

- 국내 일정 (화순·담양·지리산): `첫날..`/`둘쨋날..` 표기, `[숙박- ...+조식]` 표기, `*...*` 안내문
- 일본 일정 (야마가타/니가타): `<제N일>` 표기, `[기본 여행 일정]` 헤더(무시), 숙박이 `[OO온천호텔]`처럼 '숙박' 글자 없이 대괄호만으로 표기됨
- 다음카페 게시글 캡처: 상단 제목줄에 날짜 범위 포함 (예: `7월4(토)~9/야마가타&...`) → 출발일 추출
- **OCR 노이즈 대응 (실사용 버그에서 학습)**: 영문·일본어 혼용 이미지를 한글 모델로 읽으면
  숫자 쓰레기(`03545017` 등)가 생김 → OCR은 `kor+eng+jpn`으로 수행 (일본 여행 일정의
  한자·가나 지명 `玉川寺`, `村上店 Kappa Sushi` 등을 살리기 위해 jpn 필수),
  `parseScheduleText(text, {ocr:true})`로 호출하면 쓰레기 토큰 필터 + 문맥 줄 병합 적용.
  짝 없는 `[` 가 생겨도 숙박류 대괄호만 최대 3줄 병합.
  직접 붙여넣은 텍스트는 `{ocr:false}`(기본)로 필터 없이 보존할 것.

## 배포 체크리스트

1. **JS/CSS를 수정하면 반드시 세 HTML의 자산 버전 쿼리(`?v=YYYYMMDD*`)를 함께 올릴 것.**
   GitHub Pages(+모바일 브라우저) 캐시 때문에 버전을 안 올리면 사용자에게 옛 코드가 실행된다
   (실제로 수정 배포 후에도 사용자 휴대폰에서 옛 버그가 재현된 사고 있음).
2. 파서 수정 시 node 단위 테스트(위 명령) + **실제 이미지 파일로 E2E 테스트**를 할 것:
   Playwright에서 `page.route('https://cdn.jsdelivr.net/**')`로 CDN을 로컬 npm 파일
   (`tesseract.js`, `tesseract.js-core`, `@tesseract.js-data/kor|eng`)로 라우팅하면
   이 환경에서도 업로드→OCR→미리보기 전체 흐름이 돌아간다. 텍스트 주입 테스트만으로 끝내지 말 것.
3. main 병합 후 Pages 배포가 `success`인지 확인 (일시 오류로 `failure`가 나면 재트리거 필요).

## 주의

- `tesseract.js` 및 한글 traineddata는 jsdelivr CDN에서 로드됨 — 이 개발 환경에서는 프록시가
  jsdelivr를 차단하므로 OCR 테스트는 npm 패키지 `tesseract.js` + `@tesseract.js-data/kor`를
  로컬 설치해서 node로 수행할 것 (실서비스 브라우저에서는 정상 동작).
- 커밋 메시지는 기존 스타일(`feat:`, `fix:`, `style:`)을 따른다.
